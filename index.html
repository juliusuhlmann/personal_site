But how does the LLM predict the next token?

An inference perspective on LLM architecture and why memory bandwidth is the inference bottleneck.

Jun 25, 2026 8 min read

By now it has propagated to the mainstream mind that LLMs are essentially "just" next token prediction machines. Context in, first token predicted, second token predicted, repeat until the end of white-collar work is achieved.

Most explanations of how LLMs work however, only cover the training part. 3Blue1Brown will tell you all about backpropagation with animations worthy of a Pixar movie, but the training-perspective drowns out the part where the prompt goes in and tokens come out. The forward pass. Inference.

Furthermore, the inference perspective is frankly better suited for explaining LLM architecture than the training one. And yet I have not found any good explanations of that perspective so … here we are.

In this essay/blog/whatever, I will attempt to convey an intuitive sense of what each part of the LLM does when predicting the next token. You'll understand why memory bandwidth is the actual bottleneck at inference and if things go well, you'll close this tab thinking: that wasn't so complicated after all.

The Full Picture. In Brief.

Enough preamble. Let's predict the next token.

Say, this is our example interaction:

|
User

How many r's are in strawberry?

LLM

The number of r's is

<user_start> How many r 's are in strawberry ? <user_end> <model_start> The number of r 's is

The goal now is to predict the next token. Probably a number. Hopefully "three," because if not, the screenshot will show up on your X feed in about 20 minutes.

Now, if you have read anything about LLMs before (or if you have scrolled LinkedIn recently and still have your sanity) you might know that each token has an embedding. The short explanation is: An embedding is a vector of numbers, capturing the meaning of a token, where similar tokens live closer together in vector space.

The textbook example uses two dimensions because humans can't visualize more than three, and can barely manage three. In reality these vectors have thousands of dimensions, which is less intuitive but considerably more useful.

The embedding of " is" in GPT-3 might be (yes the space is included):

is =

Each token in the interaction above has such an embedding of size 12,288. The goal now is to get from those floating point numbers to a probability distribution over the next token.

The way we will get there is this: take the embedding of the last token (" is") and pass it through a sequence of layers, each adding information without changing the vector's size. Some of those layers gather context from previous tokens; others process the current vector in isolation. Either way, we come out the other side with a vector still of size 12,288, which we'll project back into vocabulary space. Out comes a probability distribution. The token with the highest probability wins (usually).

I get that starting out with the last token's embedding seems weird, but LLMs house so many layers that the output vector bears about as much resemblance to its starting point as today's Elon does to the Elon of 2014.

So if we simplify that process one more time, at the risk of becoming reductionist: map the last token to embedding space, do some very expensive linear algebra which sometimes involves the previous tokens, but always keeps the vector at the same size, and map it back to tokenland to get a probability distribution over the next token. If the final resulting vector is somewhere between "three" and "3" we might predict (30% " three", 25% " 3", 10% " two" ...).

Two Steps. Repeat Until Sentient.

Here is the dirty secret of the transformer architecture: there are basically two things happening, and then they happen again and again and again. GPT-3 has 96 blocks. Each one does the same two things: Neural Network and Attention.

Between the embedding and unembedding steps, GPT-3 runs the same four operations over and over again. Two of those steps are normalization: for every number in your 12,288-dimensional vector, subtract the mean and divide by the standard deviation of all the numbers. The "Feed forward" step is just a regular neural network layer. That leaves the attention step, the only place in the architecture where tokens actually talk to each other, and the part we shall focus on here.

Our 12,288-dimensional embedding of " is" (we'll call "input vector") enters Multi-Head Attention, which runs 96 attention heads in parallel. All of these heads receive our input vector. 96 attention heads in each multi-head attention layer, of which we have 96, makes for 96 × 96 = 9,216 heads in total.

These heads are a large part of the reason why your Claude Code subscription now makes up a substancial part of your monthly expenses. But before we get lost in a debate about whether AI is net-cheaper than developers and how badly we need to get off the AI wagon hurtling toward doom, let's rather skip the doomerism and zoom in on what such an attention head actually does. The process looks like this:

We derive a query from our input vector and all tokens can "attend" to that query with their keys.

In the example above "Subject" attends to the query "Are any of you guys subjects?". If a key of a given previous token matches our query, we add its value information to our input vector. For example, the information that a "number"(meaning a specific quantity of something) was just mentioned is added.

Queries, keys, and values are all 128-dimensional vectors, entirely determined by the previous tokens themselves and the model parameters living in our current attention head. If a query and key point in a similar direction, we do some computation on the value vector (128 doesn't go into 12,288) and then add it to the input vector. The more similar the direction, the more is added.

(Saying "key of a previous token" is a bit of a simplification as the deeper the layer, the more each key and value reflects the entire context rather than any single token. I'll call it "key of a previous token" because "a key vector that originated from token X but has since accumulated further information from every token that came before it" would undermine the very understanding it's trying to convey, so I'm keeping the lie)

Attention is simple

Now, the full attention mechanism works as follows:

  1. Calculate the query, key and value vector from our input vector
  2. Fetch the keys and values from all previous tokens
  3. Compare each key against our query
  4. The more a key matches the query, the more of its value information is added to our input vector

All 96 heads result in their own query and contribute their tweak to the input vector. Out the other side comes a vector still of size 12,288, enriched with context from the previous tokens. It then passes through two normalizations and one neural network step, before entering the attention stage again in the next block with different queries, different keys, and different values.

All of this requires those key and value vectors to be accessible. The process of storing and fetching key and value vectors is called KV-caching. Fetching from memory is much more efficient than recalculating them every time. The problem is how many numbers there are to fetch. What I described above was one head in one multi-head attention layer. GPT-3 has 96 blocks, each with 96 heads running in parallel, each demanding the key and value vectors of every previous token in the context. Multiply that out if you like. Or skip to the next section, which has already done it for you.

So why is memory bandwidth the bottleneck?

A single H100 (the GPU behind most LLM deployments) stores 80GB of data in High-Bandwidth Memory (HBM). The actual compute units, however, can only do math on data loaded into SRAM, which holds a maximum of 117MB. At some point, the key and value vectors must be loaded from HBM to SRAM. The speed of that transition (called "memory bandwidth") in an H100 is about 3.35 terabytes per second.

This might sound like a lot, but let's do the numbers for GPT-3 and then extrapolate to larger models.

GPT-3 has 96 transformer blocks, each with 96 attention heads. The maximum number of previous tokens on which an LLM bases its next-token prediction is called the context window. To predict the next token at full context, we need to fetch 2 × 96 × 96 × context_size key and value vectors (key and value vectors for all previous tokens for all heads).

With GPT-3's context window of 2,048 tokens and a head dimension of 128, that's 2 × 96 × 96 × 2,048 × 128 ≈ 4.8 billion numbers. In FP16 that's 9.6 gigabytes of data fetched to predict a single token! Considering that context windows are now 100× larger and models have additionally grown in size, you can see that memory bandwidth becomes the bottleneck quicker than you can say "Spare me AI overlords".

As you can see, the KV-cache scales linearly with context length. Compute grows too, but memory bandwidth usually hurts first: more context means more KV-cache data to fetch.

We could compare the H100 to a Ferrari on the autobahn and the KV cache to a traffic jam. However, I'm just as sick of bad AI metaphors as you are, and most of them are misleading anyway. So I'd rather just save your time and end this essay right here.

One more thing. Remember to clear your context windows.
