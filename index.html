But how does the LLM predict the next token?

An inference perspective on LLM architecture and why memory bandwidth is the inference bottleneck.

Jun 20, 2026 3 min read

By now it has propagated to the mainstream mind that LLMs are essentially "just" next token prediction machines. Context in, first token predicted, second token predicted, repeat until the end of white-collar work is achieved.

Most explanations of how LLMs work however, only cover the training part. 3Blue1Brown will tell you all about backpropagation with animations worthy of a Pixar movie, but the training-perspective drowns out the part where the prompt goes in and tokens come out. The forward pass. Inference.

Furthermore, the inference perspective is frankly better suited for explaining LLM architecture than the training one. And yet I have not found any good explanations of that perspective so … here we are.

In this essay/blog/whatever, I will attempt to convey an intuitive sense of what each part of the LLM does when predicting the next token. You'll understand why memory bandwidth is the actual bottleneck at inference and if things go well, you'll close this tab thinking: that wasn't so complicated after all.

The Full Picture. In Brief.

Enough preamble. Let's predict the next token.

Say, this is our example interaction:

|
User

How many r's are in strawberry?

LLM

The number of r's is

<user_start> How many r 's are in strawberry ? <user_end> <model_start> The number of r 's is

The goal now is to predict the next token. Probably a number. Hopefully "three," because if not, the screenshot will show up on your X feed in about 20 minutes.

Now, if you have read anything about LLMs before (or if you have scrolled LinkedIn recently and still have your sanity) you might know that each token has an embedding. The short explanation is: An embedding is a vector of numbers, capturing the meaning of a token, where similar tokens live closer together in vector space.

The textbook example uses two dimensions because humans can't visualize more than three, and can barely manage three. In reality these vectors have thousands of dimensions, which is less intuitive but considerably more useful.

Each token in the interaction above has such an embedding of size 12,288. The goal now is to get from those floating point numbers to a probability distribution over the next token.
